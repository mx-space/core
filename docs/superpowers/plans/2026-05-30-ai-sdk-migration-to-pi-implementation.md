# AI SDK Migration to @earendil-works/pi-ai — Implementation Plan

Date: 2026-05-30
Status: Generated by planning workflow wf_90613940-f6f (5 verify scouts → plan synth → 3-lens review → finalize)
Companion spec: `docs/superpowers/specs/2026-05-30-ai-sdk-migration-to-pi-design.md`

## Preamble

Replace mx-core's hand-rolled OpenAI/Anthropic runtime with a single `PiRuntimeAdapter` backed by `@earendil-works/pi-ai` in one atomic PR, while preserving every external contract callers depend on: the public `IModelRuntime` surface (expanded additively, then trimmed at step-7), the byte-exact public SSE envelope consumed by api-client/Shiroi/Yohaku, the leader/follower Redis stream wire format inside `AiInFlightService`, and all admin chat behaviour. The riskiest moves are the TypeBox port of `ai.prompts.ts` schemas (step-4) — a silent loss of a Zod `.refine` or field name corrupts structured outputs across summary/spam/translation — the `PiRuntimeAdapter` itself (step-5) where pi event translation, content-block ordering, usage shape, and the `validate: false` opt-out must exactly match the existing pipeline of `callWriter`/`callEditor`, the `ai_agent_conversations` drop-and-recreate (step-9) which is destructive and breaks rolling-deploy expand-contract by explicit user agreement, the new SSE controller protocol (step-12) which must wire `reply.raw.on('close', ...)` to abort pi mid-stream AND survive socket-closed write races, and the admin transport rewrite (step-18) which bypasses `@haklex/rich-agent-core`'s `TransportAdapter`. Step ordering: legacy runtimes survive until step-7, which runs AFTER step-8 swaps all call sites to the new interface, so the codebase compiles continuously. The pgEnum reduction (step-3a) uses expand-contract CREATE TYPE NEW + USING cast + DROP OLD because Postgres cannot drop enum labels in place. Invariants that MUST NOT regress: (1) the byte-exact public SSE envelope `event: token\ndata: <chunk>\n\n` / `event: done\n\n` (no `data:` line) / `event: error\ndata: {"message":"..."}\n\n`, (2) the cached-hydrate path that fires a whole serialised model as a single `token` payload after `sentToken===false` with pinned case-transform behaviour, (3) the leader-side Redis XADD field encoding `(type, JSON.stringify(data))`, (4) ai-summary/insights/writer `incrementTokens` side effects fire exactly once at stream-end (not on abort/error), (5) `ResponseInterceptor` `{data, meta?}` envelope plus snake_case wire conversion on every new endpoint, (6) persisted AssistantMessage round-trips through pi.complete without validation error even after abort. Tests use pi's faux provider; live suites stay as the regression net; byte-exact fixture snapshots gate every wire-touching step.

### What is the EXACT public-facing SSE token envelope contract that AiInFlightService preserves across the migration?

**Verdict:** 三事件信封：`token`/`done`/`error`，皆以 `event: <name>\n` + 可选 `data: <payload>\n` + 终止空行 `\n` 写出。`token.data` 流式时为裸文本 chunk（非 JSON），cached-hydrate 退路时为 `JSON.stringify(model)` 整体载荷；`done.data` 为 `undefined`，故仅 `event: done\n\n` 无 `data:` 行；`error.data` 为 `JSON.stringify({message})`。Scoping pass 之断言（cached path 发 `{type:'token', data: JSON.stringify(model)}` 后跟 `{type:'done'}`）属实，证据在三 controller 之 `sentToken` 分支与 `createImmediateDoneStream`。需注意：api-client `models/ai.ts:45-53` 将 `error.data` 标为 `string`，但实际线缆是 JSON-字符串化的 `{message}` 对象 — 此为 follower 解析时需 `JSON.parse` 的隐含约定（已存在的契约，迁移须保留 byte-for-byte）。

**Recommendation:** Step 9 之 `ai-inflight.service.spec.ts` 适配器测试当 pin 下列五条不变量，逐字节断言 reply.raw.write 序列：

(1) **Streaming-token 帧**：leader 经 `push({type:'token', data: chunk.text})` 推入 Redis stream（XADD field `data` = `JSON.stringify("chunk")`，即带引号字串），follower `parseStreamEvent` 反序列化后 `event.data` 为裸字串，controller `sendSseEvent(reply, 'token', "chunk")` 因 typeof === 'string' 走 `payload = data` 直通分支（sse.util.ts:41），线缆字节 = `event: token\ndata: chunk\n\n`（**无引号**）。

(2) **Cached-hydrate 退路**：当 `existingResultId` 命中 → `createImmediateDoneStream` 仅发 `{type:'done', data:{resultId}}`，controller 见 `sentToken===false` → 调 `getXById(resultId)` → `sendSseEvent(reply, 'token', doc)`，doc 为对象走 `JSON.stringify(doc)` 分支，线缆字节 = `event: token\ndata: {"id":"…",…}\n\n` 紧接 `event: done\n\n`。

(3) **Done 帧**：始终 `sendSseEvent(reply, 'done', undefined)` → 字节 = `event: done\n\n`（**无 `data:` 行**，sse.util.ts:34-39 分支）。

(4) **Error 帧**：`sendSseEvent(reply, 'error', {message})` → `event: error\ndata: {"message":"…"}\n\n`（含 controller catch 与 inflight push 'error' 两路径，二者同形）。

(5) **顺序与终结**：任一帧序列以 `done` 或 `error` 收尾后 `endSse(reply)` 关闭 raw socket；`for await` 循环遇 done|error 必 break（三 controller 一致）。

测试 fixture 当涵盖 cached / streaming / error 三场景，并对 Redis XADD field 编码（`type` 裸字串、`data` JSON.stringify 之）单独断言，因 leader→follower 之 Redis 帧亦属内部 ABI（pid 多副本场景下两副本可能各走 leader / follower 路径，须同形）。

迁移注意：(a) sse.util 之 string 直通分支不可改 — 否则 cached path 整对象与 streaming chunk 之编码会冲突；(b) `done.data===undefined` 无 `data:` 行之行为不可改 — api-client 类型已声明 `data: undefined`；(c) `error.data` 实际为 JSON-字符串化对象但 api-client 类型标为 `string` — 此 drift 应在 spec 内显式记录或同步修类型，**不可静默改变线缆**。

### Where exactly do admin model registry calls need to land?

**Verdict:** Add a new endpoint GET /api/ai/registry/models?providerId={piProviderId} on AiController; do NOT extend the existing /ai/models, /ai/models/list, or /ai/models/:providerId routes. Those three are live-fetch endpoints that hit a configured provider's REST API (requires apiKey + runtime.listModels()), whereas the registry lookup is a static catalog read from pi-ai's built-in registry (no apiKey, no live HTTP, independent of any saved AIProvider row). Keeping them separate avoids overloading semantics and matches the spec's stated rationale of server-mediating the pi registry so pi-version drift between admin and core can be centrally controlled.

Today's dropdown is NOT a registry: the AIProviderDrawer renders provider.defaultModel as a plain TextInput backed by an HTML <datalist id="ai-models-${providerId}-drawer">. The <option> list is populated from props.providerModels (AIProviderModel[]), which the parent settings view caches under modelCacheKey in TanStack Query. The list is filled only after the admin clicks "Fetch models" — that handler calls getModelList() in apps/admin/src/api/ai.ts which POSTs /ai/models/list with {providerId, type, apiKey, endpoint} and hits AiController.fetchModelsList -> createRuntimeForModelList -> runtime.listModels() (live OpenAI /v1/models or hardcoded Anthropic list). There is no pre-populated catalog and no static hardcoded list — it is empty until "Fetch models" is pressed. So both branches of the question are partially true: a server endpoint exists (POST /ai/models/list), but it does live provider HTTP, not registry lookup.

Request: GET /api/ai/registry/models?providerId=<pi providerId> (string, required; e.g. openai, anthropic, openrouter, deepseek). Auth: @Auth() (matches sibling AI routes). No apiKey/endpoint params — registry is static catalog. Implementation calls pi-ai getModels(providerId) and maps Model<Api> entries into a UI subset.

Response shape (camelCase in TS, snake_case on the wire per ResponseInterceptor): bare envelope { data: RegistryModel[] } where RegistryModel = { id: string, name: string, contextWindow?: number, maxTokens?: number, inRegistry: true, costs?: { inputPerMillion?: number, outputPerMillion?: number, cachedInputPerMillion?: number } }. inRegistry is fixed true so the admin form can branch on registry-hit vs registry-miss (per spec: hide contextWindow/maxTokens inputs when hit, show them as editable on miss). Return [] if providerId is unknown to pi (not 404 — admin renders an empty combobox plus free-text fallback).
