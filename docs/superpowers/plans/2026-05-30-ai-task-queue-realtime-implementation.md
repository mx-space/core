## Preamble

Spec 2 layers realtime fan-out, per-language concurrency, structured-output streaming, and USD cost capture on top of the spec-1 pi-runtime migration. The riskiest part is the streamStructured implementation in PiRuntimeAdapter (incremental JSON parse of toolcall_delta buffers) combined with the per-segment push contract widening — get this wrong and the markdown live preview keeps working but the lexical typewriter stays silent. Do-not-break invariants: (a) spec-1's public SSE envelope contract for the admin agent transport (AiAgentSseEvent token/done/error) remains byte-identical AND public SSE controllers (/ai/summary, /ai/insights, /ai/translation) MUST filter out new 'partial' events at the wire boundary; (b) all PiRuntimeAdapter additions stay backward compatible — generateText / generateStructured / streamMessage / generateTextStream signatures and return shapes unchanged; (c) MappedUsage.cost remains optional, callers that ignore it keep working; (d) Redis task hash keeps its existing fields, only additive HSET (totalCost cents-as-string, optional in type), pre-spec-2 task hashes that lack totalCost still parse cleanly as cost: undefined; (e) socket fanout via @socket.io/redis-adapter stays untouched, only new BusinessEvent enum + helper added; (f) admin TanStack Query keys (adminQueryKeys.ai.*) stay byte-identical, only payload interior gains optional cost. Cancellation semantics under p-limit must preserve the existing error.name === 'AbortError' rethrow pattern AND must not let orphaned workers fire late progress/log emits after the task is cancelled. The phase enum is FROZEN at 8 values: created|started|progress|status|log|result|stream|deleted. Source-of-truth files are pinned: BusinessEvents.AI_TASK_UPDATE lives in apps/core/src/shared/constants/event.constant.ts; AiTaskUpdatePayload lives in apps/core/src/processors/task-queue/task-queue.types.ts; the admin mirror in apps/admin/src/socket/types.ts is a hand-duplicated literal (no monorepo import). Rolling-deploy invariant: the LUA_RECOVER_STALE return-shape change MUST stay backward compatible — old callers reading Number(result) must keep working.

### What does the PiRuntimeAdapter expose today for streaming, and what additional surface does spec 2's lexical typewriter UI need?

**Verdict:** PiRuntimeAdapter today: generateText, generateStructured (TSchema, validate?:boolean), generateTextStream (yields {text}), streamMessage (returns pi AssistantMessageEventStream verbatim — emits text_delta, thinking_delta, toolcall_start/delta/end, done, error frames), listModels — and streamStructured EXISTS as a method but THROWS 'streamStructured reserved for spec 2' (placeholder). cost/usage is captured on every method via mapUsage → {promptTokens, completionTokens, totalTokens, cost: pi usage.cost.total ?? 0}; for generateStructured cost is accumulated across iterations into usageAccum. The IModelRuntime interface already declares streamStructured?<T extends TSchema>(opts: GenerateStructuredOptions<T>) => AsyncIterable<StructuredStreamChunk<Static<T>>> where StructuredStreamChunk<T> = {partial: Partial<T>, delta?: string, done?: boolean, final?: T} — so the contract is in place. Markdown strategy: streaming wired through streamMessage (NOT generateTextStream) — iterates pi events, on text_delta calls await push({type:'token', data: delta}); the pre-migration push hook is preserved and threaded via TranslationStrategyOptions.push (typed as AiStreamEvent = token/done/error union). Base-strategy callWriter/callEditor: today call generateStructured with validate:false then normalise + Value.Check; NO streaming — output arrives as one shot. Lexical strategy: translate() destructures options WITHOUT push — only onToken/signal/existing/reviewerRuntime/metrics; callWriter is invoked from translateAllUnits and never pushes per-segment partials. Spec 2 must: (1) implement PiRuntimeAdapter.streamStructured by consuming pi stream()'s toolcall_delta events, JSON-parsing the accumulated tool-call argument buffer on each delta, yielding {partial: Record<string,unknown>, delta: rawChunk} and a terminal {partial: final, done: true, final}; (2) widen TranslationStrategyOptions or add a new perSegment push so lexical strategy can stream — currently lexical destructure has no `push`; (3) thread the partial-JSON stream from callWriter through to a per-segment dispatcher (callWriter today is fully buffered — needs an overload or sibling streamWriter that drives streamStructured); (4) define the partial-JSON contract: {partial: {translations: {[unitId]: string | {[memberId]: string}}, sourceLang?: string}} matching the writer schema, so the admin can diff segment-by-segment as translations[unitId] keys appear; (5) markdown strategy already pushes token deltas (line 89) — only needs the worker-side buffer at the task-queue layer to coalesce.

**Recommendation:** Spec 2 implementation order: (a) flesh out PiRuntimeAdapter.streamStructured to consume pi stream() events — accumulate toolcall arg-buffer on toolcall_delta, attempt incremental JSON parse (jsonrepair-tolerant), yield {partial, delta} per delta, yield {partial: final, done: true, final} on toolcall_end + validate if validate!==false; (b) extend TranslationStrategyOptions.push semantics or add a new push variant carrying {lang, segmentId, partial}; (c) add LexicalTranslationStrategy passthrough of options.push into callWriter, and add a sibling callWriterStreaming in base-strategy that uses runtime.streamStructured?.() and falls back to current callWriter when adapter lacks streaming; (d) markdown path already works — no strategy change needed, only task-queue stream-buffer wiring; (e) keep streamMessage as the chat/agent transport (unchanged); (f) drop the reserved-throw in PiRuntimeAdapter once streamStructured lands and remove it from the `?` optional in the interface to make stream support mandatory.

Abort semantics under p-limit: `context.signal` is a real `AbortSignal` (TaskExecuteContext.signal, task-queue.types.ts L84) threaded into `generateTranslation(..., signal)`. The pi runtime already aborts the underlying stream when the signal fires. Pattern for the rewrite:
  1. `throwIfAborted(context.signal)` (or `this.checkAborted(context)`) at the TOP of the limited worker, BEFORE the `generateTranslation` call — guarantees a cancelled task never spends a slot on a redundant call.
  2. In the per-language `try/catch`, re-throw when `error.name === 'AbortError'` (already the current pattern at L243); swallow + `failedCount++` for any other error.
  3. Wrap the outer `await Promise.all(languages.map(l => limit(...)))` in a try/catch ONLY if you need to distinguish "task aborted" from "some langs failed". Promise.all rejects fast on the first AbortError thrown inside a `limit(...)` worker; queued (not-yet-started) lang workers will still execute their `throwIfAborted` guard and throw immediately, so they consume no LLM calls. The first AbortError bubbles up; outer caller (TaskQueueProcessor) recognises `error.name === 'AbortError'` and marks the task `Cancelled` instead of `Failed`.
  4. Sequencing detail: `progress` updates must happen via an atomic `++done` counter (NOT loop index `i`), as the spec sketch already shows (L215-220).
  5. Token counting is concurrency-safe — `context.incrementTokens` is Redis `HINCRBY` (noted at spec L229).

Rewrite sketch (pseudocode):

```ts
import pLimit from 'p-limit'
import { throwIfAborted } from '~/utils/abort.util'

private async executeTranslationTask(payload, context) {
  this.checkAborted(context)
  const aiConfig = await this.configService.get('ai')
  const languages = resolveTargetLanguages(payload.targetLanguages, aiConfig.translationTargetLanguages)
  if (!languages.length) { await context.appendLog('warn', 'No target languages specified'); return }

  await context.updateProgress(0, 'Starting translation', 0, languages.length)

  const concurrency = Math.max(1, Math.min(10, aiConfig.translationLangConcurrency ?? 3))
  const limit = pLimit(concurrency)
  const translations: Array<{ translationId: string; lang: string; title: string }> = []
  let done = 0
  let failedCount = 0

  await Promise.all(languages.map((lang, i) => limit(async () => {
    throwIfAborted(context.signal)            // skip work if cancelled before slot opened
    await context.appendLog('info', `Translating to ${lang} (${i + 1}/${languages.length})`)
    try {
      const result = await this.generateTranslation(payload.refId, lang, context.incrementTokens, context.signal)
      translations.push({ translationId: result.id, lang: result.lang, title: result.title })
    } catch (error: any) {
      if (error.name === 'AbortError') throw error // bubble out of Promise.all → caller marks Cancelled
      failedCount++
      await context.appendLog('error', `Failed to translate to ${lang}: ${error.message}`)
    } finally {
      const cur = ++done
      await context.updateProgress(Math.round((cur / languages.length) * 100),
        `Translated ${cur}/${languages.length}`, cur, languages.length)
    }
  })))

  await context.setResult({ translations })
  if (failedCount === languages.length) context.setStatus(TaskStatus.Failed)
  else if (failedCount > 0) context.setStatus(TaskStatus.PartialFailed)
}
```

## Rollback

All changes are git-revert-safe at the step boundary because each step ships as one commit (per the spec-1 workflow). To roll back: `git revert <sha>..<sha>` over the spec-2 commit range — there are no destructive schema migrations, no database column changes (storage is Redis-only and additive), and no API contract narrowing. Redis state cleanup notes: (1) `task-queue:room-subs:*` SET keys carry 5-minute TTLs and self-expire — no manual cleanup needed even if the rollback leaves stale subscribers; (2) the `totalCost` HSET field on each task hash is additive AND typed as optional, so old/new parseTask both handle its absence — leaving it behind after rollback is harmless; (3) the LUA_RECOVER_STALE script change (step-6a) returns `{count, ...ids}` — old callers that did `Number(result)` on the array will break, so the step-6a callers update is non-optional. To roll back step-6a safely: revert step-6a, step-6, step-7 together as a unit (the LUA script and ALL its callers must move in lockstep). Frontend rollback: `mx-admin:ai-task-stream` is a window CustomEvent — no subscribers means no-op; revert of SocketBridge restores polling-only behaviour (but at the spec-2 30s interval until step-21 is also reverted — recommend reverting steps 20+21+22 together to restore the 5s polling default). Spec-1 SSE envelope contract (token/done/error) is never narrowed in spec 2 — even partial rollbacks of streamStructured / lexical strategy leave the agent transport intact; the public SSE 'partial' filter added in step-12 is purely defensive and harmless if reverted (the widened union just stops being filtered, but no public consumer was emitting it pre-spec-2).
