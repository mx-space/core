# Release Notes Narrative — Design Spec

**Date**: 2026-05-14
**Scope**: `.claude/skills/release-core/SKILL.md` + `.github/workflows/release.yml`
**Status**: Approved (brainstorm)

## Problem

The mx-core server release pipeline currently produces GitHub Release notes by running `npx changelogithub` in CI. The output is a faithful list of conventional commits grouped under `### Features` / `### Bug Fixes`, with each entry being the raw commit subject and a short sha link. This format serves developers reading the changelog but does not communicate user-facing value: a reader cannot tell from the release notes what a feature does, why it matters, what behaviour changed, or what (if anything) they must do to upgrade.

The release pipeline must keep producing a machine-friendly changelog file for developers (existing `apps/core/CHANGELOG.md` in Angular preset), but the GitHub Release body should switch to a human-narrative format authored by the release-driving agent.

## Goals

1. GitHub Release body for `apps/core` (server pipeline) becomes a narrative document with a TL;DR, optional Breaking Changes / Upgrade Notes, narrative Highlights, and human-described change entries.
2. The narrative is generated by the agent running the `release-core` skill, locally, before the release commit is created — not in CI, not via an external LLM call from CI.
3. The user reviews the generated narrative as a gate before the release commit is made.
4. `apps/core/CHANGELOG.md` retains its Angular-preset format unchanged — it serves developers and tooling.
5. `packages/api-client` release pipeline is untouched (it has no GitHub Release today).

## Non-goals

- Changing the api-client release flow.
- Replacing or removing `apps/core/CHANGELOG.md`.
- Generating release notes from CI (would require API keys in CI and would not be reproducible locally).
- Translating release notes (English only, matching current convention).

## Solution

### Pipeline change (server release, `apps/core`)

Insert a new step **6.5 — Generate release notes** into the `release-core` skill, between existing step 6 (CHANGELOG) and step 7 (commit):

1. Agent reads the unreleased commits with `git log v<prev>..HEAD --no-merges --pretty='%H %s%n%b' -- apps/core` to obtain full subjects and bodies (the latter to surface `BREAKING CHANGE:` footers).
2. Agent selects the **main entries** from commits with types `feat:` / `fix:` and any `BREAKING CHANGE:` / `!` markers — matching the conventional-changelog scope used today.
3. Agent additionally scans `refactor:` / `chore:` / `perf:` / `docs:` commits and selects only those whose body or subject indicates a user-visible effect (dependency major bump, behaviour change, performance improvement, public API tweak). Non-user-visible ones are skipped.
4. Agent decides between **simple** and **full** structure (see "Output format" below) based on commit count and category.
5. Agent writes the rendered markdown to `apps/core/RELEASE_NOTES.md`, overwriting any previous content. The file is committed in the release commit, so each tag's notes are reproducible from the tagged tree.
6. Agent prints the rendered notes to the chat and asks the user to approve or request edits. The user MUST approve before the skill proceeds to step 7. If the user requests edits, the agent revises and re-prompts.

### Commit scope change (step 7)

The release commit is currently `git add apps/core/package.json apps/core/CHANGELOG.md`. It becomes:

```bash
git add apps/core/package.json apps/core/CHANGELOG.md apps/core/RELEASE_NOTES.md
```

Commit message and `--no-verify` semantics unchanged.

### CI change (`.github/workflows/release.yml`)

In the `build` job:

- Add an explicit guard step before the Release step. `softprops/action-gh-release@v3` does not hard-fail on a missing `body_path` file (it falls back to an empty body), so an explicit check is required to enforce the "no curated notes → no release" rule:

  ```yaml
  - name: Verify release notes present
    run: |
      if [ ! -s apps/core/RELEASE_NOTES.md ]; then
        echo "::error::apps/core/RELEASE_NOTES.md is missing or empty. The release-core skill must produce it before tagging."
        exit 1
      fi
  ```

- Update the `Release` step (uses `softprops/action-gh-release@v3`) to read the body from the committed file:

  ```yaml
  - name: Release
    uses: softprops/action-gh-release@v3
    with:
      tag_name: ${{ github.ref_name }}
      name: ${{ github.ref_name }}
      body_path: apps/core/RELEASE_NOTES.md
      prerelease: ${{ contains(fromJSON('["alpha", "beta"]'), github.ref_name) }}
      files: release-linux.zip
  ```

- Delete the trailing `Changelog` step that runs `npx changelogithub` — it would otherwise overwrite the body after the action-gh-release step runs.

### Output format

#### Full structure

Used when commits ≥ 4, or when any `feat:` / breaking change is present.

```markdown
## TL;DR

<One sentence summarising the version's core change and user impact, 15–25 words.>

## Breaking Changes  ← only rendered when present

- **<area>**: <what changed + why>. **Migration**: <action the operator must take>.

## Highlights

<2–3 paragraphs of prose, each ~40–80 words. One topic per paragraph. Describe
user-visible behaviour and value, not commit subjects. Mention notable
constraints or follow-ups.>

## Changes

### Features
- <Human-readable description of what users can now do, e.g. "Admins can view
  enrichment cache details, screenshots, and probe results via new
  `/admin/enrichment` endpoints."> ([#PR] or [sha])

### Bug Fixes
- <Human-readable description, e.g. "Screenshot fallback chain now prefers
  `og:image` when present instead of always rendering."> ([#PR] or [sha])

### Other  ← only when user-visible refactor/chore/perf was selected
- <Human-readable description>

## Upgrade Notes  ← only when manual operator action is required

<env var / migration / config changes; cite exact commands or file paths>

---

**Full Changelog**: https://github.com/mx-space/core/compare/v<prev>...v<this>
```

#### Simple structure

Used when commits ≤ 3 AND no `feat:` AND no breaking change.

```markdown
## TL;DR

<One sentence.>

## Changes

- <Human-readable description.> ([#PR] or [sha])

---

**Full Changelog**: https://github.com/mx-space/core/compare/v<prev>...v<this>
```

### Authoring rules for the agent

These are embedded in the `release-core` skill as a self-discipline checklist:

1. Use user-facing language. Translate commit subjects into what the user can observe or do. "Refactor pool" → "Connection pool now reuses sockets across requests, reducing handshake latency."
2. Do not copy commit subjects verbatim into entries; rewrite them.
3. Link priority: PR number (`#2708`) > issue number > commit sha (short 7-char + URL). Prefer PR links when the commit message contains `(#NNNN)`.
4. Every `Breaking Changes` entry MUST include a `**Migration**:` clause with a concrete action — even if the action is "no action required, just observe the new behaviour".
5. TL;DR is one sentence, 15–25 words. It must name the headline change and its user impact.
6. Highlights paragraphs are 40–80 words each; aim for 2–3 paragraphs total in the full structure.
7. Structure auto-selection:
   - If `any feat:` OR `any breaking change` is selected → **full**.
   - Else if ≥ 4 entries are selected → **full**.
   - Else if all selected entries are internal-only (chore/dependency bumps with no user-visible effect) → **simple** with the "Internal maintenance release" TL;DR wording (see rule 9), regardless of count.
   - Otherwise → **simple**.
8. Empty sections (no breaking, no upgrade-required, no user-visible refactor) are omitted, not rendered with placeholder text.
9. When the release is genuinely internal-only (all selected entries are chore/dependency bumps without user effect), TL;DR states this plainly (e.g. "Internal maintenance release; no user-facing changes."), Highlights is omitted, and the `Other` section lists the chore items.

### User review gate (step 6.5 end)

After writing `apps/core/RELEASE_NOTES.md`, the agent prints the full rendered file to chat and asks:

> "This is the proposed release notes body for v<X.Y.Z>. Approve to proceed to the release commit, or request edits."

The agent does NOT advance to step 7 without an affirmative response. If the user requests edits, the agent revises the file and re-prompts. This gate cannot be skipped silently — the skill text states it as a hard requirement.

### Edge cases

| Case | Behaviour |
|------|-----------|
| All selected commits are chore/refactor with no user-visible effect | TL;DR says "Internal maintenance release; no user-facing changes." Simple structure; `Changes` lists the chore items. |
| Single commit | Simple structure; one TL;DR sentence + one Changes line. |
| Breaking change in a minor release | Full structure; `Breaking Changes` section rendered at the top (above Highlights). |
| User edits `RELEASE_NOTES.md` after the agent writes it but before the release commit | Edits are committed as part of the release commit; CI reads the user-edited version. No divergence possible. |
| CI fails after tag push and a patch release follows | New release runs the same flow; `RELEASE_NOTES.md` is overwritten. Previous release's notes remain on the GitHub Release page (immutable record). |
| Need to fix release notes after the tag is already published | Run `gh release edit vX.Y.Z --notes-file apps/core/RELEASE_NOTES.md` (or `--notes "..."`) to update only the notes body. Tag and assets are untouched. Add this row to the skill's Rollback table. |

### Rollback path

If the narrative approach turns out to be unworkable, restoration is straightforward:

1. In `release.yml`: drop `body_path` and `fail_on_unmatched_files` from the Release step; restore the `Changelog` step that runs `npx changelogithub`.
2. In `SKILL.md`: remove step 6.5 and revert step 7's commit scope to two files.
3. `RELEASE_NOTES.md` becomes a stale file; it can be left in the repo as a historical artefact or deleted.

No data loss; no tag changes required.

## Out of scope

- `packages/api-client` pipeline (no GitHub Release today, untouched).
- Localisation (English only).
- Replacing `apps/core/CHANGELOG.md` with narrative content.
- Automating release-notes generation from CI without human review.

## Affected files

- `.claude/skills/release-core/SKILL.md` — add step 6.5, update step 7 commit scope, update Rollback table, add authoring rules section.
