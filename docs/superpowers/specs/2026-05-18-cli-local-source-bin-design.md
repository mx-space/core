# CLI Local Source Bin Design

Date: 2026-05-18

## Summary

The CLI development workflow must make bare `mxs` resolve to the local
workspace source implementation when the developer is working inside
`packages/cli`. This must happen without a manual global symlink step, without
requiring developers to remember `pnpm dev`, and without adding development
directory detection to the published global CLI.

The accepted approach is to make repository-local command resolution provide
`packages/cli/node_modules/.bin/mxs`, which points at the package bin shim. The
bin shim then prefers the TypeScript source entry when it exists and falls back
to the built distribution entry otherwise.

## Goals

- In `mx-core/packages/cli`, `mxs` resolves to the repository-local CLI source.
- The source command executes through `tsx src/bin/mxs.ts`.
- The behavior is automatic after repository installation or setup.
- Published global `@mx-space/cli` does not inspect `cwd` to redirect into a
  development checkout.
- Development source execution defaults to the local development endpoint.
- The ongoing config simplification remains intact.

## Non-Goals

- Do not add `cwd`-based development redirects to the published CLI shim.
- Do not require manual global symlink management.
- Do not restore removed profile config fields such as `api_base`, `auth_base`,
  or `client_id`.
- Do not publish or release a package as part of this change.

## Command Resolution

```text
┌──────────────────────────────┐
│ shell: mxs                   │
└───────────────┬──────────────┘
                ▼
┌──────────────────────────────┐
│ ./node_modules/.bin/mxs      │
│ generated by repo setup      │
└───────────────┬──────────────┘
                ▼
┌──────────────────────────────┐
│ packages/cli/bin/mxs.cjs     │
└───────────────┬──────────────┘
                ▼
        ◆ source exists? ◆
          /            \
         ▼              ▼
┌────────────────┐  ┌──────────────────┐
│ tsx src/bin/   │  │ node dist/bin/    │
│ mxs.ts         │  │ mxs.mjs           │
└────────────────┘  └──────────────────┘
```

The shell should find the repository-local `.bin` entry before any global
installation. The repository must therefore ensure that
`packages/cli/node_modules/.bin/mxs` exists and points to the package shim.

This resolution path intentionally does not depend on replacing the user's
global `mxs`. Global installations may remain installed; they simply should not
win when the developer is inside the CLI package and the local `.bin` directory
is on `PATH`.

## Bin Shim Behavior

`packages/cli/bin/mxs.cjs` owns the package-level dispatch:

- If `src/bin/mxs.ts` exists, execute it with `node --import tsx`.
- Otherwise, execute `dist/bin/mxs.mjs`.
- If neither entry exists, fail with an actionable build/setup message.

This keeps the logic package-local. A published tarball contains `bin` and
`dist`, but not the TypeScript source entry, so published CLI execution naturally
uses `dist`. A repository checkout contains `src`, so development execution
naturally uses source.

## Local Dev Profile Behavior

Source execution is the development signal. When `src/bin/mxs.ts` is the active
entry and no explicit target is provided, the CLI should default to the local
development profile.

| Condition | Default profile | API base |
| --- | --- | --- |
| Source execution, no explicit profile or URL | `local-dev` | `http://localhost:2333` |
| Source execution with `--profile` or `MXS_PROFILE` | Explicit profile | Resolved from that profile |
| Source execution with `--api-url` or `MXS_API_URL` | No default profile | Resolved from the explicit URL |
| Distribution execution | No development default | Existing config behavior |

The local development default must take precedence over the current global
profile pointer. This prevents a production profile from silently winning during
repository-local CLI development.

Explicit user intent still has the highest priority:

- `--profile`
- `MXS_PROFILE`
- `--api-url`
- `MXS_API_URL`

## Config Shape

The profile config simplification remains part of the design. Profile config
should not persist legacy derived fields:

- `api_base`
- `auth_base`
- `client_id`

The resolver may derive runtime `apiBase`, `authBase`, and `clientId`, but these
values are runtime outputs, not persisted profile config fields.

In development default mode, `auth login` should write credentials and the
simplified config under `local-dev`.

## Implementation Units

| Unit | Responsibility |
| --- | --- |
| Local bin setup | Create or maintain `packages/cli/node_modules/.bin/mxs` pointing to the package shim |
| Bin shim | Prefer source when `src/bin/mxs.ts` exists; otherwise use distribution output |
| Source entry | Enable the development default profile only during source execution |
| Config resolver | Resolve `local-dev` before the current profile when development default mode is active |
| Auth login | Persist login results to the resolved `local-dev` profile in development default mode |
| Tests | Cover local bin/source behavior, dev profile precedence, explicit overrides, and profile guard behavior |

## Validation

Manual validation from `packages/cli`:

```text
which mxs
→ .../packages/cli/node_modules/.bin/mxs

readlink node_modules/.bin/mxs
→ ../bin/mxs.cjs or an equivalent repository-local shim

NODE_DEBUG=child_process mxs --version
→ node --import tsx .../packages/cli/src/bin/mxs.ts --version
```

Automated validation:

```text
pnpm -C packages/cli exec vitest run test/core/config-store.test.ts test/bin/mxs.test.ts test/commands/auth/login.test.ts
pnpm -C packages/cli run typecheck
```

## Risks

- The local `.bin` entry only wins if the user's shell places local
  `node_modules/.bin` before global installation directories. This repository
  already assumes that convention in the reported environment.
- If the local `.bin` entry is created by install hooks, install scripts must be
  idempotent and must not mutate global package-manager state.
- The ignored `packages/cli/bin` directory means the package shim's tracking
  status must be handled deliberately if future implementation changes require
  committing it.

## Acceptance Criteria

- From `packages/cli`, bare `mxs` resolves to the repository-local command.
- The resolved command runs `tsx src/bin/mxs.ts`.
- Published global CLI behavior remains free of development checkout detection.
- Source execution defaults to `local-dev` and `http://localhost:2333` when no
  explicit profile or URL is supplied.
- Explicit profile and URL inputs override the development default.
- Profile config remains simplified and does not reintroduce legacy derived
  fields.
